Microservices Design Patterns using Clean Architecture - Top Interview FAQs

1. Why do we need Design Patterns in Microservices?

Answer:

  • Reduce code duplication.
  • Solve recurring software problems.
  • Improve maintainability.
  • Increase scalability.
  • Promote loose coupling.
  • Improve testability.

Example: Order Service communicates with Shipping Service using Publish-Subscribe instead of direct API calls.

3. Why is CQRS used?

Issue: Read and Write operations have different performance requirements.

Solution:

  • CreateOrderCommand
  • UpdateOrderCommand
  • GetOrderQuery
  • GetOrderHistoryQuery

Benefit: Better scalability and optimized queries.

4. Why do we use MediatR (Mediator Pattern)?

Without MediatR

Controller
      |
Service
      |
Repository

With MediatR

Controller
      |
Mediator
      |
Handler
      |
Repository

Purpose: Loose coupling and cleaner architecture.

5. Why Repository Pattern instead of Entity Framework directly?

Issue: Business layer depends on EF Core.

Repository solves:

  • Loose coupling
  • Easy Unit Testing
  • Replace database easily
  • Cleaner business logic
6. What problem does Saga Pattern solve?

Scenario

  1. Create Order
  2. Reserve Inventory
  3. Process Payment
  4. Create Shipment

If Payment fails, Inventory Reservation should rollback.

Saga performs distributed transaction with compensation.

7. Why Outbox Pattern?

Problem:

Database saved successfully but Kafka publish failed.

Solution:

  • Save Order
  • Save Event into Outbox table
  • Background Worker publishes event later

No event loss.

8. Why Event-Driven Architecture?

Instead of

Use

 |    |     |

All services become independent.

9. Why Strategy Pattern?

Issue: Shipping cost changes based on delivery type.

Strategy Example
Standard ₹50
Express ₹150
Overnight ₹300

Runtime selects the correct algorithm.

10. Why Factory Pattern?

Create objects without exposing creation logic.


ShippingFactory.Create("FedEx")

ShippingFactory.Create("DHL")

ShippingFactory.Create("UPS")
11. Why State Pattern?

Order moves through different states.


Pending

↓

Paid

↓

Packed

↓

Shipped

↓

Delivered

Each state controls allowed operations.

12. Why Specification Pattern?

Business rules become reusable.

Example

  • Customer Active?
  • Credit Available?
  • Stock Available?
  • Address Valid?
13. Why Adapter Pattern?

Different courier APIs have different request formats.

Courier Adapter
FedEx FedExAdapter
DHL DHLAdapter
BlueDart BlueDartAdapter

Application uses one common interface.

14. Why Circuit Breaker?

Problem: Shipping API is down.

Circuit Breaker

  • Stops repeated API calls.
  • Returns fallback response.
  • Protects the system.
15. Why Retry Pattern?

Handles temporary failures.

Example

  • Retry 3 Times
  • 2 Seconds Delay
  • Then Fail
16. Why Cache Aside Pattern?

Reduce database load.

↓

↓

Not Found

↓

↓

17. Why Dependency Injection?

Objects should not create their dependencies.

Instead of


new OrderRepository()

Use


IOrderRepository repository

Improves testing and loose coupling.

18. Explain complete Order → Shipping flow using Design Patterns.
  1. Client → Order API (Facade)
  2. Middleware validates request
  3. Mediator sends CreateOrderCommand
  4. Handler executes CQRS
  5. Factory creates Order Aggregate
  6. Specification validates business rules
  7. Repository saves data
  8. Unit of Work commits transaction
  9. Outbox stores event
  10. Kafka publishes OrderCreated
  11. Inventory reserves stock
  12. Saga manages transaction
  13. Shipping Factory selects courier
  14. Adapter calls external API
  15. Retry + Circuit Breaker handle failures
  16. State changes Order to Shipped
